# Architecture & Gotchas

How the capture + triage core fits together, and the non-obvious decisions that
break silently if changed. Read this before modifying any asset.

## Data flow

```
User clicks 🐛
  → widget captures viewport (html2canvas) + console logs, BEFORE modal opens
  → user picks category, writes ≥10 chars, optionally adds attachments
  → POST /api/bug-reports  (JSON only — NO image bytes)
      → auth check → zod validate → insert row (display_id from trigger)
      → insert reporter as participant
      → create signed upload URLs, pre-store deterministic public URLs on row
      → respond { data, signedUploadUrl, additionalSignedUrls }
  → widget uploads image bytes DIRECTLY to Storage via uploadToSignedUrl()
Triager opens admin list
  → GET /api/bug-reports?status=... (reads bug_reports_with_details + similar_count)
  → detail page reads one report + messages + participants (via BugReportService)
  → status updates + thread replies write back through the service
```

## The signed-URL upload handshake

**This is the most important pattern. Do not "simplify" it by posting the image
in the request body.**

Why: screenshots are often 1–2MB. Posting them through a Next.js route on
Vercel hits the 4.5MB body limit and causes server-side `ECONNRESET` on large
binary writes under load. Instead:

1. The POST body carries only `wants_screenshot`, `screenshot_format`, and
   `additional_image_count/_formats` — flags, not bytes.
2. The route calls `supabase.storage.createSignedUploadUrl(path)` and returns
   `{ path, token, signedUrl }` for each file.
3. The browser calls `supabase.storage.uploadToSignedUrl(path, token, file)` —
   bytes go straight to Storage, never through the server.
4. Because the storage path is deterministic (`<reportId>/screenshot.jpg`), the
   route pre-computes `getPublicUrl(path)` and stores it on the row *before* the
   upload finishes. The URL resolves the moment the upload completes.

Trade-off: if the browser upload fails after the row is created, the row keeps a
`screenshot_url` pointing at a missing object (a broken image, not a crash). The
report itself still saves. This is intentional — capturing the report text +
logs matters more than the screenshot.

## display_id sequence

`display_id` (BUG-NNNNNN) is generated by a Postgres **SEQUENCE** via a
`BEFORE INSERT` trigger — never `SELECT MAX(display_id)+1`. The original used
MAX()+1 and saw an ~87% failure rate under concurrent submits (a gap of 2,062
between 306 real reports and a max id of 2368 proved the race). The API route
*also* retries up to 3× on the `bug_reports_display_id_key` unique violation as a
backstop. Keep both.

## The details view

`bug_reports_with_details` joins reporter identity (and optionally
institution/department names) onto each row. The triage list and detail read
**from the view**, so the client never joins identity itself. When you add
columns to `bug_reports`, the view's `b.*` picks them up automatically — but if
you add a *joined* field you must edit the view.

## similar_count dedup

The GET route computes `similar_count` per page of results: it extracts a
normalized signature from each report's `console_logs` (first error line, with
UUIDs → `UUID`, `:line:col` → `:L:C`, hex → `0xADDR`), counts how many reports
share it, and reports `count - 1` as "others like this." It is computed, not
stored — so it reflects only the current result page, which is good enough for
triage grouping and avoids a stored-dedup maintenance burden.

## Gotchas

- **Dynamic Tailwind classes**: the widget builds class names like
  `border-${color}-500`. Tailwind's JIT only emits classes it can see as
  complete strings at build time, so these get purged in production. Either
  hardcode the three category color sets, or add them to `safelist` in
  `tailwind.config`. (The original MyJKKN widget hardcoded each category block.)
- **Mount once, client-side, inside auth**: `<BugReporterWidget />` must render
  on authenticated pages only (it calls `supabase.auth.getUser()` server-side on
  submit). Mount it in the authenticated layout, not the root layout.
- **`admin = true` = service-role**: `BugReportService` methods that read across
  all reporters pass `admin: true`, which must resolve to a SERVICE-ROLE client
  on the server. Never import the service into a client component — call the API
  routes from the browser instead.
- **Storage MIME policy must match the widget**: the bucket's
  `allowed_mime_types` is the server-side guard. If you widen the widget's
  accepted file types, widen the bucket array in the same change, or uploads 400.
- **RLS off = nothing reads**: the migration enables RLS but ships no policies.
  Until you add them (see SKILL.md → "Define access control"), only the
  service-role key can touch the tables — reporter-facing reads return empty.
- **Screenshot privacy**: `html2canvas` captures whatever is on screen,
  including any PII visible at report time. If your app shows sensitive data,
  add elements to the ignore list (`data-html2canvas-ignore`) or mask before
  capture.
